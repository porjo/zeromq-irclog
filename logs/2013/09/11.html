[00:01]pieterh 1. elliptic curve cryptography vs. other options
[00:01]pieterh "The best assured group of new public key techniques is built on the arithmetic of elliptic curves." -- NSA, at http://www.nsa.gov/business/programs/elliptic_curve.shtml
[00:01]bear_ security shall over ZeroMq
[00:01]pieterh 2. Curve25519 vs other elliptic curves
[00:01]pieterh - no interference by NSA/NIST
[00:01]pieterh - fastest known curve
[00:01]pieterh - no known patents on the technology
[00:01]pieterh http://cr.yp.to/ecdh/patents.html
[00:01]pieterh 3. CurveZMQ as such
[00:01]pieterh - perfect forward security between endpoints
[00:01]pieterh - fast enough for serious use
[00:01]pieterh - simple, you cannot make configuration errors
[00:01]bear_ message just message
[00:01]pieterh 4. ZeroMQ security model (in ZMTP/3.0 and in forthcoming 0MQ/4.x)
[00:01]pieterh - extensible to ANY security mechanism (e.g. DTLS if you want it)
[00:01]pieterh negatives:
[00:01]bear_ encrypted Data as message too
[00:01]pieterh yes
[00:01]pieterh With libzmq you can do security over one hop (e.g. pub to sub)
[00:01]pieterh with libcurve you can do end-to-end security over untrusted middlemen
[00:01]pieterh (libcurve is a separate library, reference implementation of CurveZMQ)
[00:01]pieterh I'll write this up on a blog post / FAQ later
